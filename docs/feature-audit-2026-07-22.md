# WA-B Solutions — Unified Feature-Wishlist Audit

*Synthesized from six independent area audits: Command Center Dashboard, Guided Setup/Onboarding, Lifecycle Automation, Payments & Checkout Confidence, WhatsApp First-Run Experience, Trust & Polish.*

---

## 1. Executive summary

The backend (`src/`) is consistently ahead of both client surfaces it feeds: Express/Postgres routes for delivery tracking, refunds, accounting, onboarding, structured categories, bundles, and inventory are implemented correctly, but the Flutter mobile app — the merchant's stated primary tool — only calls a fraction of them, and even where it does, `mobile/wab_app/lib/screens/order_detail.dart` alone discards three payloads (`history`, `refunds`, `payment_attempts`) the backend already sends it on every `GET /api/orders/:id`. Two structural inversions repeat across nearly every report: the mobile app is systematically thinner than the secondary web dashboard (`public/dashboard.html`) — onboarding, accounting/payouts, rider/ETA/refund actions, cart-nudge/loyalty config, categories, and bundles exist only there — and WhatsApp, the flagship channel, gets a materially weaker conversational experience than Instagram/Messenger (natural-language quantity parsing is wired only for IG/Messenger, so "I want 2 fried rice" silently drops to qty 1 on WhatsApp but parses correctly on Instagram). Of the roughly 50 sub-features audited, about a third are fully DONE, a third are PARTIAL (backend exists, client wiring doesn't), and a third are genuinely MISSING with no code anywhere (win-back, post-purchase review, delivery feedback, back-in-stock alerts, verified-shop badge, auto-retry, payment reminders). **The single biggest cross-cutting risk is `payment_status` correctness**: the mobile "mark paid" quick action (`order_detail.dart` → `PATCH /api/orders/:id/status`) only ever writes `orders.status`, never `payment_status`, so a merchant recording a cash sale in the app will see the chip flip to "paid" while it is silently excluded from today's GMV (`home.dart`), analytics (`analytics.routes.js`), and loyalty — this is shared infrastructure (`order.service.js`'s `updateOrderStatus` vs. `markOrderPaid`) that several planned features (payment timeline, settlement reports, "send reminder") would otherwise be built on top of incorrectly. Separately, a genuine production bug was found — the birthday-coupon cron is scheduled only in `src/worker.js`, which PM2 never starts (`deploy/ecosystem.config.js` runs only `wa-saas-api`) — meaning one already-"DONE" feature is silently dead in production today; this should raise the bar on verifying "done" claims before further work is greenlit on faith.

## 3. Cross-area integration risks

**1. Mobile app is a systematically thinner client than the backend — the single largest and most repeated finding.** Every one of the six reports independently found the same pattern: rider assignment/delivery/refund actions (Command Center, Trust & Polish), the entire onboarding checklist (Onboarding), accounting/payout/settlement views (Payments), cart-nudge and loyalty configuration plus broadcast audience targeting (Lifecycle Automation), and structured categories/bundles/featured flags (Command Center) all exist correctly on the backend and in `public/dashboard.html`, but have **zero** client in `mobile/wab_app/lib`. Given the product owner's framing that the mobile app is primary, this is not six separate gaps but one systemic one: the web dashboard is quietly the "real" full-featured client today.

**2. `order_detail.dart` is the single highest-leverage file — three different reports independently flag it for missing wiring to the same backend response.** `GET /api/orders/:id` already returns `history`, `refunds`, and `payment_attempts` (`order.routes.js:288-301`) — the code comment literally says this is "for the dashboard" — but `order_detail.dart:35` reads only `res['order']`. Command Center additionally flags missing rider-assignment UI (`PATCH /api/orders/:id/delivery`) and notes/refund actions; Trust & Polish flags the same delivery route plus `PATCH /api/orders/:id/estimates` and `POST /api/orders/:id/refund`; Payments flags the discarded `history`/`refunds`/`payment_attempts` and the missing receipt link. **These should be built as one coherent screen redesign, not four incremental patches** — they share the same fetch call, the same screen, and the same underlying order object.

**3. `payment_status` is set inconsistently, and multiple planned features would inherit the bug if built now.** Command Center's Integration Risk #1 and Payments' Integration Risk #4 describe the same root cause from two angles: the mobile "mark paid" action only writes `orders.status` via `updateOrderStatus` (`order.service.js:442-461`), never `payment_status`, never stock, never loyalty — while `home.dart`'s GMV/paid tiles and nearly every `analytics.routes.js` metric gate on `payment_status = 'paid'`. There is no mobile-reachable endpoint that calls the correct `markOrderPaid` (`order.service.js:744-874`) at all today. Any future "payment timeline," "daily settlement report," or "send payment reminder" feature built before this is fixed will display or act on incorrect payment state.

**4. Delivery-lifecycle events never proactively notify the customer, and this shows up as a gap in three areas independently.** Trust & Polish (`setEstimates` has no notification call; `notifyRiderAssigned` texts the rider, not the customer), Lifecycle Automation ("delivery feedback" is entirely missing despite the `notifyOrderStatusChange` hook already existing), and WhatsApp First-Run (no address memory compounding the "fast reorder" journey) all point at the same underlying hook point (`src/services/notification.service.js`'s `notifyOrderStatusChange`) as the natural, currently-underused extension point.

**5. Failed-payment reason data is captured at the gateway layer but destroyed before it reaches any surface that could show it.** Payments' Integration Risk #2 (MTN/Paystack `reason`/`gateway_response` dropped in `webhook.processor.js` before calling `conversation.handler.js#handlePaymentFailure`, whose signature doesn't even accept a reason) is a prerequisite for Payments' "failed-payment troubleshooting" MISSING item, but it's also relevant to WhatsApp First-Run (the customer-facing failure message is generic) and to any future mobile order-detail failure display (Command Center/Trust & Polish territory). Fixing any client UI for this without first threading the reason through the backend would just add a UI with nothing to show.

**6. `client.dart`'s lack of a service layer means every "does X call Y" audit had to be done by grepping call sites, and this itself is a risk going forward.** Payments' Integration Risk #3 notes `mobile/wab_app/lib/api/client.dart:36-46` is a thin generic `get/post/patch/delete` wrapper with no endpoint-specific methods — there is no single place documenting which backend routes the app actually uses. This compounds risk #1 above: as more endpoints get wired in the phases below, without an endpoint registry/service layer, gaps like these will keep recurring silently rather than being caught by a type-checked contract.

**7. Two "primary vs. secondary" inversions exist in parallel, and both should be named explicitly to the product owner as a pattern, not treated as two unrelated facts.** Mobile app vs. web dashboard (risk #1) and WhatsApp vs. Instagram/Messenger conversational quality (WhatsApp First-Run's Integration Risk #1: `nl.intent.js`'s free-text NLU is deliberately IG/Messenger-only per its own docstring, so WhatsApp customers get a materially worse typed-text experience than Instagram customers on the exact same phrasing). Both mean the *secondary* surface currently has the *better* experience, inverted from stated product priority.

**8. A "DONE" backend feature can still be dead in production — this is a verification-process risk, not just a code risk.** The birthday-coupon job (Lifecycle Automation) reads as fully complete in the codebase and is configurable in the web dashboard, but is scheduled only in `src/worker.js`, which the PM2 process list (`deploy/ecosystem.config.js`) never starts. This means "grep shows the code exists" is not sufficient evidence of "the feature works" — a deployment/process-list check should be part of any future "is this DONE" verification, not just a code-presence check.

## 4. Recommended phased roadmap

Sequenced so shared backend/data-correctness issues land before UI is built on top of them, quick wins land early, and repeated work on the same file happens once per file rather than in scattered passes.

### Phase 0 — Backend correctness fixes (prerequisite for almost everything downstream)
**Includes:** (a) Fix the birthday-cron deployment gap — move the schedule into `src/server.js`'s `startCronJobs()` or stand up a real worker process in `deploy/ecosystem.config.js`. (b) Resolve "mark paid" semantics — decide whether the cash-payment action should route through a `payment_status`-aware equivalent of `markOrderPaid` (stock/loyalty/GMV side effects), then fix `order.service.js`/`order.routes.js` accordingly. (c) Thread the gateway failure `reason` through `webhook.processor.js` → `conversation.handler.js#handlePaymentFailure` → an updated i18n template placeholder, so it exists for later UI/message surfacing.
**Why first:** These are shared building blocks — Phase 1's payment timeline, Phase 4's failure-reason display, and Phase 2's "Paid" tile all depend on `payment_status` and failure data being correct. Building UI on top of the current bug would bake it in further.
**Files:** `src/server.js`, `deploy/ecosystem.config.js`, `src/services/order.service.js`, `src/routes/order.routes.js`, `src/services/webhook.processor.js`, `src/services/conversation.handler.js`, `src/utils/i18n.js`.

### Phase 1 — Order detail screen consolidation
**Includes:** One redesign pass on `order_detail.dart` adding: rider assignment (`PATCH /api/orders/:id/delivery`), ETA (`PATCH /api/orders/:id/estimates`), refund action (`POST /api/orders/:id/refund`), payment timeline (rendering the already-returned `history`/`refunds`/`payment_attempts`), receipt link (view/copy/share), merchant notes, and the corrected "mark paid"/"record cash payment" action from Phase 0. Also humanize `payment_status` labels in `widgets/common.dart`'s `StatusChip` ("Awaiting payment" / "Payment received") consistently across `home.dart`, `order_detail.dart`, and anywhere else it's rendered.
**Why second:** Three separate reports (Command Center, Payments, Trust & Polish) flagged this exact file for the same underlying data the backend already sends it — doing it once avoids three rounds of churn on the same screen.
**Files:** `mobile/wab_app/lib/screens/order_detail.dart`, `mobile/wab_app/lib/widgets/common.dart`, `mobile/wab_app/lib/api/client.dart` (add endpoint-specific methods here rather than ad-hoc `get/post/patch` calls — addresses cross-area risk #6).

### Phase 2 — Command Center completion: Today view, notifications, catalog structure
**Includes:** Low-stock tile (`/api/inventory/reorder-suggestions`), notification bell feed (`/api/notifications`), conversation summary in `chat.dart`, structured categories/featured-flag/bundles UI in `products.dart` (new categories screen). "New customers today" and "messages needing reply" need **new backend aggregate queries first** (neither exists) — flag as backend work, not just wiring.
**Why third:** Independent of Phases 0-1; high visible impact on the dashboard's core promise ("Today view" is the spec's first bullet); backend for most of it already exists.
**Files:** `mobile/wab_app/lib/screens/home.dart`, `chat.dart`, `products.dart`, new categories/notifications screens; backend additions to `src/routes/order.routes.js` or `customer.routes.js` for the two missing aggregates.

### Phase 3 — Guided onboarding on mobile
**Includes:** New checklist screen consuming `/api/onboarding/status`, `/webhook-health`, `/test-message`, `/sample-catalog`; expand `settings.dart` to cover logo/banner/slug/industry (already supported by `business.routes.js`); surface the admin incomplete-setup list. Explicitly **rename/clarify** the existing `onboarding.dart` (a first-run marketing carousel, unrelated) to avoid confusion with the new checklist screen.
**Why here:** Confirmed to be pure UI/client work — same `requireAuth('any')` + `resolveBusinessId` pattern already used successfully elsewhere, no auth rework needed — making it a genuinely low-risk, high-impact phase once Phases 0-2 have stabilized the patterns being reused (client methods, screen conventions). The "test MoMo payment" and "go-live checklist" sub-items are blocked on product decisions (see §5) and should be scoped out of this phase until answered.
**Files:** new `mobile/wab_app/lib/screens/onboarding_checklist.dart`, `settings.dart`, `lib/api/client.dart`, an admin screen for the incomplete-setup list.

### Phase 4 — Payments UX and accounting surfacing
**Includes:** Merchant payout summary and daily settlement report screens (`/api/accounting/payouts`, `/payout-balance`, `/daily-sales`, `/reconciliation`); failed-payment reason display (now unblocked by Phase 0); "send payment reminder" (new backend endpoint + mobile button — cadence/consent rules need a product decision, see §5); auto-retry deferred pending a product/compliance decision.
**Why here:** Depends on Phase 0's failure-reason threading; naturally follows Phase 1's order-detail work since it extends the same screen family.
**Files:** new `src/routes` addition for payment reminders, new mobile accounting/payouts screens, `order_detail.dart` (failure-reason display, reminder button).

### Phase 5 — Lifecycle automation build-out
**Includes:** Reorder reminder, win-back, post-purchase review, delivery feedback, back-in-stock alert. Recommend building a generic `automations`/`campaign_templates` table + registry (per the source report's explicit suggestion) rather than six more bespoke cron functions copy-pasted from `cart_nudge`/`loyalty_birthday` — this avoids compounding the maintenance burden the audit already flagged. Also wire mobile broadcast/promo audience targeting (`broadcasts.dart`, `promos.dart`) so merchants can actually use segment/tag/min-spend targeting the backend already supports.
**Why here:** Entirely new features, independent of Phases 0-4; sequenced after the order-lifecycle notification hook (`notifyOrderStatusChange`) has been touched in Phase 4, since post-purchase-review and delivery-feedback attach to it.
**Files:** new `src/services/automations/*`, new migration for an automations table, `src/utils/audience.js`, `mobile/wab_app/lib/screens/broadcasts.dart`, `promos.dart`.

### Phase 6 — WhatsApp conversational fixes
**Includes:** Fix the WhatsApp quantity-drop bug (extend quantity pre-parsing to WhatsApp before `tryTypedProductAdd`, not just IG/Messenger); add a `TRACK`/`MY ORDER` keyword to the global command block and NLU vocab; add a `customers.address` column and reuse it in `reorderLastOrder`/checkout; add delivery-fee NLU; replace `featured`-flag popularity ranking with real order-volume-based ranking.
**Why here:** Self-contained to `conversation.handler.js`/`nl.intent.js`, independent of the mobile phases; sequenced later only because it's lower urgency than payment-correctness and dashboard completeness, not because of dependencies.
**Files:** `src/services/conversation.handler.js`, `src/services/nl.intent.js`, `src/utils/helpers.js`, `src/models/migrate.js` (new `address` column), `src/services/order.service.js` (popularity query).

### Phase 7 — Trust & polish finishing pass
**Includes:** Customer-facing — branded receipts (pull `logo_url`), expose rider phone in `receipt.routes.js`'s response, refund/cancellation policy copy, verified-shop badge (blocked on a policy decision, see §5), image-optimization pipeline (blocked on a dependency-approval decision), real share-to-WhatsApp. Merchant-facing — search on products/inbox/broadcasts/promos, inline validation everywhere, loading skeletons (needs a new package approval), audit-trail mobile view; bulk actions/undo as lower-priority nice-to-haves.
**Why last:** Lowest risk of blocking other work, highest proportion of items requiring a product decision or a new dependency before engineering can start.
**Files:** `src/routes/receipt.routes.js`, `public/receipt.html`, `src/models/migrate.js` (verified-shop column), various mobile screens for search/validation/skeletons.

## 5. Open questions for the product owner

1. **"Mark paid" semantics** — should the mobile cash-payment action always run the full `markOrderPaid` side-effect pipeline (stock decrement, loyalty, GMV counting), or is there a legitimate product reason cash payments should be tracked differently from MoMo payments? This blocks Phase 0/1.
2. **Auto-retry failed MoMo payments** — is genuine unattended auto-retry desired (re-charging a customer's MoMo without their re-initiation), given the consent/compliance implications, or should this stay customer-initiated with a better retry prompt? This is a compliance decision, not just engineering effort.
3. **"Send payment reminder"** — automatic (e.g., X hours after an order is stuck pending) or merchant-triggered on demand? What cadence/opt-out rules apply, given SMS/WhatsApp opt-out is already enforced elsewhere in the codebase?
4. **Live MoMo test payment during onboarding** — is a real sandbox/live 1-pesewa charge acceptable (cost, sandbox availability, UX of "did it work"), or should the "payment setup" onboarding step stay a configuration-presence check, as it is today?
5. **"Verified shop" badge** — what's the verification process and criteria (manual admin review, auto-derived from onboarding completion, KYC document check)? This needs a policy decision plus new schema before any engineering starts.
6. **Product image handling** — is a real upload/resize pipeline (new dependency, e.g. `sharp`/`multer`) in scope, or should this stay "paste a URL" with only lazy-loading/`srcset` added as a lighter partial fix? The current paste-URL approach was noted as an explicit product decision in the code comments ("no in-app photo upload yet") — confirm whether that's still the intended scope.
7. **Lifecycle automation architecture** — is a generic reusable "automation templates" engine (more upfront design work, avoids the copy-paste sprawl the audit warns about) preferred over the faster bespoke-per-feature approach that mirrors the existing `cart_nudge`/`loyalty_birthday` pattern, given the desire for things to work "smooth and without integration issues" long-term?
8. **Mobile vs. web dashboard scope** — should the mobile app become the single source of truth for all merchant actions (effectively deprecating equivalent flows in `public/dashboard.html`), or is the web dashboard intentionally being kept as a permanent "power user" surface for some flows (onboarding, accounting, rider/refund actions)? This determines whether every PARTIAL/MISSING mobile item in this report needs a mobile equivalent built, or whether some gaps are acceptable to leave web-only.
9. **Proactive customer delivery notifications** — should ETA changes and rider assignment push a WhatsApp message to the customer automatically (new plumbing), or is the current model (customer must reopen a previously-texted receipt link to see updates) acceptable?
10. **Holiday offers** — what counts as a "holiday" in the target market, and how should eligible dates/discount rules be configured per shop? Needs a small content/policy spec before Phase 5 engineering.

## Summary

The merchant mobile dashboard (`mobile/wab_app/lib/screens/{home,orders,order_detail,inbox,chat,products,customers,analytics}.dart`) covers the basics of each of the five required sections and, where it does call the backend, the endpoints and field names line up correctly — `admin_home.dart`'s and `home.dart`'s stat tiles match `admin.routes.js`/`order.routes.js` output field-for-field, and `analytics.dart` consumes `/api/analytics` exactly as the backend returns it (the product owner's claim about analytics richness is verified true). ## Integration risks

1. **"Mark paid" quick action is semantically wrong.** `order_detail.dart`'s status chips include `'paid'` from `orderStatuses` (`orders.dart:14-16`) and call `PATCH /api/orders/:id/status` (`order_detail.dart:41-48`). But `updateOrderStatus` (`order.service.js:442-461`) only ever writes the `status` column — it never touches `payment_status`, never decrements stock, never runs loyalty logic, and never triggers the stock/low-stock side effects that `markOrderPaid` (`order.service.js:744-874`) performs. Meanwhile `home.dart`'s "Paid" tile and "Sales today" GMV are computed from `payment_status = 'paid'` (`order.routes.js:55-56`), and `analytics.routes.js` gates almost every metric on `payment_status = 'paid'` too. A merchant tapping "paid" in the app to record a cash sale will see the order chip flip to "paid" but it will **not** count toward today's GMV/paid-order stats, won't decrement stock, and won't trigger loyalty — a real, hard-to-notice data-integrity gap between what the UI implies and what actually happens. There is no mobile-reachable endpoint that calls `markOrderPaid` at all (it's only invoked from the webhook path via `conversation.handler.js`).

2. **Rider assignment and delivery tracking are backend-complete but mobile-invisible.** `PATCH /api/orders/:id/delivery` (`order.routes.js:326-372`) supports `rider_name`/`rider_phone`/`delivery_status`/`delivery_proof_url` and fires a WhatsApp message straight to the rider (`notification.service.js:331-359`). No screen or button in `mobile/wab_app/lib` calls this route — a merchant cannot assign a rider from the app despite it being explicitly named as a required quick action.

3. **Low-stock, reorder suggestions, and margins (`inventory.routes.js`) have zero mobile client.** Grep across `mobile/wab_app/lib` for any `/api/inventory` call returns nothing. This is the direct data source the Today view's "low-stock products" requirement needs and it's unused.

4. **The dashboard notification/bell feed (`/api/notifications`, `notification.routes.js`) — which explicitly aggregates new orders, failed payments, low stock, and "talk to a human" handoff events — has no mobile client at all.** The app relies solely on ephemeral FCM push (`shell.dart:29-67`) for these signals; if a push is missed or the app was closed, there is no in-app place to review it, even though the backend already persists and serves this exact feed.

5. **Structured categories, bundles, featured flag, variants, and add-ons are fully built on the backend (`category.routes.js`, `product.routes.js:270-448` and `580-757`) but have no mobile UI whatsoever** — category is only a free-text field on the product form (`products.dart:219-220`), disconnected from the `categories` table the backend maintains for sort order/hidden-in-menu state.

6. **Conversation summary endpoint (`conversations.routes.js:117-149`) is unused** despite `chat.dart` polling the thread every 8 seconds (`chat.dart:37`) — a natural place to surface it.

7. Nothing broken was found in the routes that **are** wired up (Today stats, orders list/detail status update, inbox/chat send+pause/resume, product CRUD + quick edit, analytics) — field names, auth (`Bearer` token via `Session.api`, consistent across all screens), and response envelopes (`success`/`orders`/`products`/etc.) all match cleanly between `lib/api/client.dart` callers and the Express routes.

### Integration risks

1. **WhatsApp drops typed quantities that aren't in "NxItem" form — Instagram doesn't.** `nl.intent.js`'s `detectIntent` (the module that strips filler like "I want"/"can I get" and extracts bare `"2 fried rice"` quantities via the regex at `nl.intent.js:138-140`) is invoked from exactly one call site: `conversation.handler.js:955`, gated by `if (['instagram', 'messenger'].includes(customer.channel) ...)` at `:936`. WhatsApp never reaches this code path. Consequently, when a WhatsApp customer types `"I want 2 fried rice"` (whether from idle or mid-browse), it goes straight to `tryTypedProductAdd` (`:1594-1620`) with the raw string. `parseQuantityExpression` (`helpers.js:288-299`) requires a literal `x`/`×`/`*` separator and does not match this phrase, so `explicit` is `null`; the whole phrase is then fuzzy-matched against the catalog via `fuzzyMatchProducts`, which matches on **substring containment** (`fuzzyMatch.js:98`: `term.includes(name)` — the full typed sentence contains the product name "fried rice"), scoring 80 and adding the item — but at `quantity: 1` (`conversation.handler.js:1607`, `explicit ? explicit.quantity : 1`), silently discarding the customer's stated "2". On Instagram/Messenger the same phrase is first rewritten to canonical `"2x fried rice"` by the `PRODUCT` case at `:974-976` before it ever reaches `tryTypedProductAdd`, so the quantity survives. Net effect: identical customer input yields correct behavior on the secondary channel and a silent under-order on the primary/flagship channel — worth fixing given WhatsApp is the product's main channel per `README.md`.

2. **`nl.intent.js`'s docstring itself flags the scope decision** (`nl.intent.js:1-16`): "the IG bot runs button-free... This module maps those typed phrases onto the conversation flow" — confirming this was a deliberate IG-only design, not an oversight, but it means the CLAUDE.md-adjacent spec bullet "Let customers type natural phrases" is materially less true for WhatsApp than for Instagram, which is the inverse of what a reviewer skimming the feature list would assume.

3. **No delivery-address memory anywhere in the schema** (`migrate.js:176-193`) means `reorderLastOrder`'s "one-tap reorder" convenience (finding 3 above, DONE) is undercut immediately afterward by a full manual address re-entry (`askForAddress` at every checkout, `:2013-2020`) — the "fast repeat order" journey isn't actually fast end-to-end.

4. **Minor, not a bug but worth noting for reviewers:** `customerOrderStatus` (`:1086`) and the order-number regex correctly scope lookups to `order.business_id === business.id && order.customer_id === customer.id` before revealing anything (`:1089`), so the "track by order number" self-service path that does exist is not an IDOR/enumeration risk — good defensive posture, just narrow in discoverability (finding 4 above).

## Integration risks

1. **Rider phone collected but never shown to the customer.** `receipt.routes.js:33` selects `o.rider_phone` from the DB but the response object at `receipt.routes.js:59-83` omits it — only `rider_name` is returned, and `receipt.html:134` only ever renders the name. If the intent was "customer can call the rider," that link is broken despite the data flowing correctly everywhere else (mobile PATCH → `order.service.js` → DB).
2. **Mobile app cannot perform three actions the customer-facing receipt depends on.** `order_detail.dart` (the mobile order screen) only calls `PATCH /api/orders/:id/status` (`order_detail.dart:44-48`). It never calls `PATCH /api/orders/:id/delivery` (rider assignment), `PATCH /api/orders/:id/estimates` (ETA), or `POST /api/orders/:id/refund` — all three exist and are wired up only in `public/dashboard.html` (`dashboard.html:1625-1726`, using the `omRiderPhone`/`omReadyAt`/`omDeliveryAt` fields and an `/orders/:id/refund` call). Since the mobile app is stated as the primary merchant tool, this means the merchant-facing polish features (rider assignment, ETA, refunds) that power the customer's trust experience are only reachable through the secondary web dashboard.
3. **Setting an ETA doesn't notify the customer.** `order.service.js:517` `setEstimates` has no accompanying notification call (contrast with `assignDelivery` at `order.routes.js:337-344`, which does fire `notifyRiderAssigned` — though that notifies the rider, not the customer, see below). A customer only sees a new ETA if they happen to reopen a receipt link they already have.
4. **Rider-assignment notification goes to the rider, not the customer.** `notification.service.js:331-345` `notifyRiderAssigned` texts the *rider* (address, customer's number to call). The *customer* is never told a rider was assigned or given the rider's identity proactively — they'd only see it by reloading the static receipt page.
5. **"Order again" vs. "share receipt" are conflated.** `receipt.html:172-177`'s only WhatsApp action sends the shop a fixed `REPEAT` message; there's no way to actually forward/share the receipt's contents to WhatsApp (no `navigator.share`, no wa.me link carrying the receipt URL/summary).
